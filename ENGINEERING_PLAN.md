# 动画教学工程规划

## 总目标

- 把当前工程从“少量模块的演示页面”升级为“可以长期承载小学信息学奥赛与小学奥数模块库的教学平台”。
- 当前已落地模块包括容斥原理、最短路线标数法、体育比赛中的学问、C++ 循环入门、C++ 排序算法、循环指挥官、诗词大赛。
- 让后续多个智能体可以并行开发不同模块，同时保持统一的教学结构、奖励机制和页面体验。

## 未来目标形态

### 1. 内容层

- 每个知识点都是一个独立模块。
- 每个模块都包含统一的页面模板：
- intro：概念引入
- rule：规则或语法
- animation：动画演示
- example：生活案例
- practice：练习
- challenge：可选闯关

### 2. 框架层

- 首页由模块注册表自动生成，不再手工硬编码导航卡片。
- 进度条由模块页面序列自动生成，不再手工维护多个页面数组。
- 练习、奖励、进度、推荐下一课由共享逻辑统一驱动。

### 3. 激励层

- 每学完一个模块可以获得星星、徽章或奖牌。
- 不同模块按阶段逐步解锁。
- 练习正确率、连续学习天数、完成数量都可以转成正向反馈。

## 迁移顺序建议

### 第 1 阶段：先建立标准

- 建立模块类型定义
- 建立模块注册表
- 建立统一教学页面协议
- 建立共享练习题数据结构
- 建立统一奖励数据结构

### 第 2 阶段：抽离共享能力

- 抽离 `LessonCode`
- 抽离 `GridDiagram`
- 抽离 `SortingBars`
- 抽离题目组件、结果反馈组件、奖励组件
- 抽离动画步骤生成器接口

### 第 3 阶段：把现有模块迁移到 modules

- 容斥原理
- 最短路线标数法
- C++ 循环
- C++ 排序

### 第 4 阶段：接入长期能力

- 学习档案
- 徽章与积分
- 学习地图
- 推荐下一课
- 错题与复习

## 多智能体协作规则

### 模块智能体

- 负责一个具体知识模块
- 交付页面、案例、练习和动画数据

### 基础设施智能体

- 负责组件库、动画引擎、模块注册表、样式体系

### 文档智能体

- 负责 README、AGENTS、TODO 路线图维护

### 质量智能体

- 负责 lint、build、页面冒烟测试、GitHub Pages 兼容检查

## 必须长期坚持的约束

- 新模块必须接入首页导航
- 新模块必须遵循统一教学页面结构
- 新模块完成后必须更新 README
- 新模块完成后必须更新 TODO 路线图状态
- 改动部署相关内容时必须确认 GitHub Pages 兼容
- 开发完成后必须先校验，再 commit 与 push
